## 0. 架构关系澄清

### MCP 协议 / Playwright MCP 服务 / DeepSeek API 三者的正确关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    LlmOrchestratorService                        │
│                    (MCP Client 角色)                              │
│                                                                  │
│  ① 将 MCP 工具定义转为 OpenAI Function Calling 格式               │
│  ② 发送 messages + tools 到 DeepSeek API                         │
│  ③ 接收 DeepSeek 返回的 tool_calls 决策                          │
│  ④ 将 tool_calls 转为 MCP 协议请求，发给 Playwright MCP Server    │
│  ⑤ 接收 MCP 执行结果，作为 tool 消息发回 DeepSeek                │
│  ⑥ 循环 ②-⑤ 直到 DeepSeek 返回最终文本                          │
│                                                                  │
│         ┌──────────────┐          ┌──────────────────┐          │
│         │  DeepSeek API │          │ Playwright MCP   │          │
│         │  (大模型)      │          │ Server (工具服务) │          │
│         │               │          │                  │          │
│         │ 角色: 推理决策  │          │ 角色: 浏览器操作  │          │
│         │ 协议: OpenAI   │          │ 协议: MCP/JSON-  │          │
│         │ Compatible     │          │ RPC 2.0 over    │          │
│         │ Function       │          │ Streamable HTTP  │          │
│         │ Calling        │          │                  │          │
│         └──────────────┘          └──────────────────┘          │
└─────────────────────────────────────────────────────────────────┘
```

**不是**「Playwright MCP 接入 DeepSeek」，而是 **「DeepSeek 通过 MCP 协议调用 Playwright 工具」**。

DeepSeek 不需要知道 MCP 协议的存在——它只理解 OpenAI Function Calling 格式。LlmOrchestratorService 在中间做**协议翻译**。

### 关键设计决策：工具调用循环在 PHP 层还是在 LLM 层？

**答案：在 PHP 层（LlmOrchestratorService）**。

```php
class LlmOrchestratorService
{
    /**
     * 工具调用循环 — 标准 MCP Agent Loop
     * 
     * @param string $providerCode  使用哪个 LLM provider
     * @param array  $messages      对话历史
     * @param array  $mcpTools      MCP 工具定义（OpenAI Function Calling 格式）
     * @param int    $maxIterations 最大工具调用轮数（防止无限循环）
     */
    public function chatWithTools(
        string $providerCode,
        array $messages,
        array $mcpTools,
        int $maxIterations = 15
    ): ChatResponse {
        $iteration = 0;
        
        while ($iteration < $maxIterations) {
            $iteration++;
            
            // ① 调用 LLM（带 tools 参数）
            $response = $this->callLlm($providerCode, $messages, $mcpTools);
            
            // ② 如果 LLM 返回普通文本（无 tool_calls），结束循环
            if ($response->isTextResponse()) {
                return $response;
            }
            
            // ③ 提取 tool_calls
            $toolCalls = $response->getToolCalls();
            
            // ④ 逐个执行工具调用
            $toolResults = [];
            foreach ($toolCalls as $toolCall) {
                // 分类路由：PHP 原生工具 vs MCP 工具
                if ($this->isPhpNativeTool($toolCall->name)) {
                    $result = $this->executePhpTool($toolCall);
                } else {
                    $result = $this->mcpExecutor->callTool(
                        $toolCall->serverName,
                        $toolCall->name,
                        $toolCall->arguments
                    );
                }
                $toolResults[] = $result;
            }
            
            // ⑤ 将工具执行结果追加到 messages
            $messages[] = $response->getAssistantMessage(); // assistant + tool_calls
            foreach ($toolResults as $result) {
                $messages[] = [
                    'role' => 'tool',
                    'tool_call_id' => $result->toolCallId,
                    'content' => $result->content,
                ];
            }
            
            // ⑥ 循环继续 — LLM 看到工具结果后决定下一步
        }
        
        throw new MaxToolIterationsExceededException($maxIterations);
    }
}
```

**这个循环在 PHP 层的原因**：
1. DeepSeek API 是无状态的——每次请求是独立的，不会自动循环
2. PHP 层控制循环次数，防止 LLM 无限调用工具（Token 成本失控）
3. PHP 层可以做审计日志（每轮工具调用都记录）
4. PHP 层可以做限流（workspace_id 级别的工具调用频率控制）

### 2.4 smart_failover 设计

```php
public function smartFailover(FailoverRequest $request): ChatResponse
{
    // ① 获取候选模型列表（按 failover_priority 排序）
    $candidates = $this->resolveCandidates($request->providerCode, $request->modelId);
    
    $attempts = [];
    $lastException = null;
    $circuitBreaker = new CircuitBreaker($request->workspaceId);
    
    foreach ($candidates as $candidate) {
        // ② 检查熔断状态
        if ($circuitBreaker->isOpen($candidate->providerCode)) {
            $attempts[] = ['provider' => $candidate->providerCode, 'status' => 'circuit_open'];
            continue;
        }
        
        // ③ 检查租户额度
        if (! $this->quotaService->hasQuota($request->workspaceId, $candidate->providerCode)) {
            $attempts[] = ['provider' => $candidate->providerCode, 'status' => 'quota_exceeded'];
            continue;
        }
        
        try {
            // ④ 尝试调用
            $response = $this->callLlm($candidate->providerCode, $request->messages, $request->tools);
            
            // ⑤ 成功 → 记录额度 → 返回
            $this->quotaService->deduct($request->workspaceId, $candidate->providerCode, $response->tokensUsed);
            $circuitBreaker->recordSuccess($candidate->providerCode);
            
            return $response->withAttempts($attempts);
            
        } catch (LlmException $e) {
            $attempts[] = ['provider' => $candidate->providerCode, 'status' => 'failed', 'error' => $e->getMessage()];
            $circuitBreaker->recordFailure($candidate->providerCode);
            $lastException = $e;
            
            // ⑥ 指数退避
            usleep(min(100000 * pow(2, count($attempts)), 4000000)); // max 4s
        }
    }
    
    throw new AllProvidersFailedException($attempts, $lastException);
}
```

### 2.5 熔断器设计（复用 Redis）

```php
class CircuitBreaker
{
    // Redis key: circuit:{workspaceId}:{providerCode}
    const FAILURE_THRESHOLD = 3;       // 3 次失败→熔断
    const COOLDOWN_SECONDS = 60;       // 60 秒冷却
    const WINDOW_SECONDS = 300;        // 5 分钟滑动窗口
    
    public function isOpen(string $providerCode): bool
    {
        $key = "circuit:{$this->workspaceId}:{$providerCode}";
        $failures = (int) Cache::get("{$key}:failures", 0);
        
        if ($failures >= self::FAILURE_THRESHOLD) {
            $openedAt = (int) Cache::get("{$key}:opened_at", 0);
            if (time() - $openedAt < self::COOLDOWN_SECONDS) {
                return true; // 熔断中
            }
            // 冷却期过，半开状态（允许一次尝试）
        }
        return false;
    }
}
```

### 3.3 McpToolExecutor 实现

```php
class McpToolExecutor
{
    private array $clients = []; // McpClient instances
    
    /**
     * 注册一个 MCP Server
     */
    public function registerServer(string $name, array $config): void
    {
        $this->clients[$name] = McpClient::connect($config['url'], [
            'transport' => $config['transport'],
            'timeout' => $config['timeout'] ?? 30,
        ]);
    }
    
    /**
     * 列出所有已注册 MCP Server 的工具
     */
    public function listAllTools(): array
    {
        $allTools = [];
        foreach ($this->clients as $serverName => $client) {
            foreach ($client->listTools() as $tool) {
                $allTools[] = new McpTool(
                    serverName: $serverName,
                    name: $tool->name,
                    description: $tool->description,
                    inputSchema: $tool->inputSchema,
                );
            }
        }
        return $allTools;
    }
    
    /**
     * 调用指定 MCP Server 的工具
     */
    public function callTool(string $serverName, string $toolName, array $arguments): McpToolResult
    {
        $client = $this->clients[$serverName] 
            ?? throw new McpServerNotRegisteredException($serverName);
        
        $result = $client->callTool($toolName, $arguments);
        
        return new McpToolResult(
            serverName: $serverName,
            toolName: $toolName,
            content: $result->content,
            isError: $result->isError,
        );
    }
}
```

## 4. ToolSchemaTranslator — 协议翻译层

这是最容易被忽略但最关键的组件。它负责两件事：

### 4.1 MCP 工具定义 → OpenAI Function Calling 格式

```php
class ToolSchemaTranslator
{
    /**
     * 将 MCP 工具的 JSON Schema 转为 OpenAI Function Calling 格式
     */
    public function toOpenAiFunction(McpTool $tool): array
    {
        return [
            'type' => 'function',
            'function' => [
                'name' => $this->normalizeToolName($tool->serverName, $tool->name),
                'description' => $tool->description,
                'parameters' => $tool->inputSchema, // MCP 已经是 JSON Schema，直接复用
            ],
        ];
    }
    
    /**
     * 工具命名规则：{server}_{tool} → 避免不同 MCP Server 的工具名冲突
     */
    private function normalizeToolName(string $server, string $tool): string
    {
        return "{$server}_{$tool}";
        // 例如: playwright_browser_navigate, playwright_browser_click
    }
    
    /**
     * 将 OpenAI tool_call 转为 MCP 调用参数
     */
    public function toMcpCall(array $openAiToolCall): array
    {
        // "playwright_browser_navigate" → server="playwright", tool="browser_navigate"
        [$server, $tool] = explode('_', $openAiToolCall['function']['name'], 2);
        
        return [
            'server' => $server,
            'tool' => $tool,
            'arguments' => json_decode($openAiToolCall['function']['arguments'], true),
        ];
    }
}
```

### 4.2 混合工具注册：PHP 原生 + MCP

```php
class AgentToolRegistry
{
    private array $phpTools = [];   // PHP 原生工具
    private array $mcpTools = [];   // MCP 工具（Phase 3 才注入）
    
    /**
     * 注册 PHP 原生工具（Phase 2 使用）
     */
    public function registerPhpTool(AgentTool $tool): void
    {
        $this->phpTools[$tool->name()] = $tool;
    }
    
    /**
     * 批量注册 MCP 工具（Phase 3 使用）
     */
    public function registerMcpTools(array $tools): void
    {
        foreach ($tools as $tool) {
            $this->mcpTools[$tool->name] = $tool;
        }
    }
    
    /**
     * 获取所有工具（OpenAI Function Calling 格式）
     */
    public function toOpenAiFunctions(): array
    {
        $functions = [];
        
        foreach ($this->phpTools as $tool) {
            $functions[] = $tool->toOpenAiFunction();
        }
        
        foreach ($this->mcpTools as $tool) {
            $functions[] = $this->schemaTranslator->toOpenAiFunction($tool);
        }
        
        return $functions;
    }
    
    /**
     * 执行工具（自动路由到 PHP 或 MCP）
     */
    public function execute(string $toolName, array $arguments, int $workspaceId): ToolResult
    {
        if (isset($this->phpTools[$toolName])) {
            return $this->phpTools[$toolName]->execute($arguments, $workspaceId);
        }
        
        // MCP 工具：从命名规则反推 server
        foreach ($this->mcpTools as $mcpTool) {
            $normalizedName = $this->schemaTranslator->normalizeToolName($mcpTool->serverName, $mcpTool->name);
            if ($normalizedName === $toolName) {
                return $this->mcpExecutor->callTool($mcpTool->serverName, $mcpTool->name, $arguments);
            }
        }
        
        throw new ToolNotFoundException($toolName);
    }
}
```

## 5. 三个辅助 MCP 的评估

### 5.1 filesystem MCP — ❌ 现阶段不装

**理由**：
- 我们的混合架构中，文件读写（Cookie 存储、llms.txt 生成、报表导出）都是 B 型代码固定流程
- `Storage` Facade 比 MCP 更高效、更安全、自带租户路径隔离
- 让 LLM 直操作文件系统引入安全风险（路径穿越、敏感文件读取）

**何时引入**：当运营人员需要自然语言指令处理文件时（如"把上周的所有复盘报告打包下载"）

### 5.2 sequential-thinking MCP — ⚠️ 可选，Phase 3 评估

**价值分析**：

对于 DeployAgent 错误分析场景：
```
Without sequential-thinking:
  LLM → "发布失败，可能是 CAPTCHA"（一次性猜测，推理过程不可见）

With sequential-thinking:
  Thought 1: 分析错误信息 → "HTTP 403 + body 含 'verify' → 特征匹配验证码拦截"
  Thought 2: 排除其他原因 → "403 非 401，排除认证过期；body 不含 'rate limit'，排除限流"
  Thought 3: 确定方案 → "建议：切换新浏览器指纹 + Playwright MCP 重试"
  Final: 明确归因 + 可执行方案
```

**但有一个关键权衡**：DeepSeek 本身推理能力极强（chain-of-thought 是其核心优势），sequential-thinking MCP 可能带来的收益有限。建议 Phase 3 跑完 DeployAgent 错误分析后，对比有无 sequential-thinking 的准确率差异，再决定是否引入。

### 5.3 context7 MCP — ❌ 生产环境不装，开发环境可用

**理由**：这是一个开发辅助工具（帮开发者查文档），不是 Agent 运行时工具。安装到开发者本地 Claude Code/Cursor 即可，无需部署到服务器。

**结论**：三个辅助 MCP，Phase 1-3 全不装。Phase 4+ 按需评估 sequential-thinking。

## 6. 与现有自研 RPA 的双轨共存

### 6.1 关键设计：两个独立后端，统一 Agent 接口

```
DeployAgent
    │
    ├── 调用 AgentToolRegistry::execute('publish_article', $args, $wsId)
    │
    ├── 路由到 PHP 原生工具 →
    │   └── RpaPublishTool::execute()
    │       └── RpaEngineClient::executeTask()  →  自研 RPA (:9901)
    │           └── 15 个成熟 JS 脚本
    │
    └── 路由到 MCP 工具 →
        └── McpToolExecutor::callTool('playwright', 'browser_navigate', ...)
            └── Playwright MCP (:8931)
                └── LLM 自适应浏览器操作
```

### 6.2 双轨决策由代码做，不由 LLM 做

```php
// AgentToolRegistry::execute() 内部
public function execute(string $toolName, array $args, int $workspaceId): ToolResult
{
    // 路由逻辑：PHP 原生工具优先（有成熟脚本的渠道）
    if ($toolName === 'publish_article' && $this->hasMatureRpaScript($args['platform'])) {
        return $this->phpTools['rpa_publish']->execute($args, $workspaceId);
    }
    
    // 无成熟脚本 → LLM 通过 Playwright MCP 自适应
    if ($toolName === 'publish_article') {
        // 不在此处路由，而是让 LLM 通过 Function Calling 自主调用 Playwright 工具
        // 此时 toolName 会是 'playwright_browser_navigate' 之类的 MCP 工具
        return $this->mcpExecutor->callTool(...);
    }
    
    // ...其他工具路由...
}
```

**路由决策在代码层，不是在 LLM 层**。LLM 不需要知道"这个渠道有没有成熟脚本"——它只知道"我可以调 `publish_article` 工具（成熟渠道会直接执行），也可以调 Playwright 浏览器工具（新渠道自适应）"。

## 7. Phase 1 最小可行实现

### 本周可交付的 LlmOrchestratorService 骨架

```php
// 只做三件事：
// 1. 统一的 chat() 接口（替代现有的分散调用）
// 2. smart_failover（替代现有的 generateContentWithModelSelection 内部逻辑）
// 3. 租户额度扣减

class LlmOrchestratorService
{
    public function chat(ChatRequest $request): ChatResponse
    {
        // Phase 1: 纯文本对话，无工具调用
        $response = $this->callLlm(
            $request->providerCode,
            $request->modelId,
            $request->messages,
            $request->options
        );
        
        // 扣减额度
        $this->quotaService->deduct(
            $request->workspaceId,
            $request->providerCode,
            $response->tokensUsed
        );
        
        return $response;
    }
    
    // callLlm() 内部：
    // 1. 从 ai_model_providers 表取 API Key（复用 ApiKeyCrypto 解密）
    // 2. 从 config 取 base_url + adapter class
    // 3. adapter->chat() 发送请求
    // 4. adapter->normalizeResponse() 标准化响应
    // 5. 返回 LlmResponse（text + tokensUsed + modelId）
}
```

## 8. 总结：最终技术决策矩阵

| 决策项 | 决定 | 理由 |
|--------|------|------|
| MCP 客户端实现 | `laravel/mcp` 官方包 | 协议完整、社区维护、代码零成本 |
| LLM 工具调用循环 | PHP 层 `while` 循环 | 成本控制、审计、限流、避免无限循环 |
| 工具路由逻辑 | 代码决断（非 LLM） | 成熟渠道走脚本、新渠道走 MCP 是规则不是推理 |
| 双轨 RPA 架构 | 自研 RPA (15脚本) + Playwright MCP | 不是替代关系，是互补关系 |
| sequential-thinking MCP | Phase 3 后评估 | DeepSeek 原生推理强，可能收益有限 |
| filesystem MCP | 不引入 | Storage Facade 更优 |
| context7 MCP | 仅开发者本地 | 开发辅助，非运行时工具 |
| 协议翻译层 | ToolSchemaTranslator | MCP JSON Schema ↔ OpenAI Function Calling |
| Phase 1 交付 | LlmOrchestratorService + DeepSeek Adapter | 最小可行，直接解决卡点 2 |
